## 1. Цель

Переделать две страницы — **«Личный кабинет»** (`/accounts/cabinet/`) и **«Мой тариф»** (`/billing/subscription/`) — в современном UX/UI, объединив их визуальный язык и компоненты. Дать пользователю одну точку входа для управления тарифом, лимитами, модулями, балансом и безопасностью.

Эталон визуала — макет `cabinet-mockup.html`. Из макета **берём:** компоновку контента (hero-баланс, KPI-полоса, лимиты с прогресс-барами, сетка планов, модули с тогглами, история операций, сессии). Из макета **НЕ берём:** боковую навигацию, тёмный сайдбар, выдуманные тарифы, выдуманные KPI-цифры.

## 2. Что НЕ трогаем

- **Горизонтальная навигация в `templates/base.html`** — остаётся как есть. Не переносим её в сайдбар, не меняем структуру меню «Рейсы / Финансы / Контрагенты / Договоры», не трогаем org-switcher и balance-badge. Любые изменения в base.html — только аддитивные (см. §3).
- **Существующие view-функции** `AccountCabinetView`, `SubscriptionView`, `subscription_upgrade`, `subscription_schedule_downgrade`, `DepositView` — логику не меняем. Можем расширить `get_context_data` дополнительными полями для UI.
- **Биллинг-движок** — `billing/services/`, `Plan`, `Subscription`, `BillingTransaction`, `Module`, `AccountModule`. Никаких изменений в моделях, миграциях, расчётах pro rata, овердрафта.
- **Тарифная сетка** — берём ровно ту, что в БД (см. §5.3). Не выдумываем «Команда» / «свой тариф» — их нет.
- **Существующие классы кнопок** (`.tms-btn`, `.tms-btn-primary` и т.д.) и правила из `docs/ui-guide.md` — соблюдаем без отступлений.

   Бэкенд для уведомлений ещё не готов → клик открывает дропдаун-заглушку с текстом «Раздел в разработке».

```

static/css/lk-tokens.css                      ← НОВЫЙ: дизайн-токены страниц ЛК
```

`lk-tokens.css` — общие переменные и компоненты (карточки, чипы, прогресс, kpi-tile), которые используются и в cabinet.html, и в subscription.html. Подключается на обеих страницах через `{% block head %}`. Префикс CSS-классов нового UI — **`.lk-`** (личный кабинет), чтобы не пересекаться с глобальными `.card`, `.section` и т.п. из других страниц.

## 5. Содержимое страниц

### 5.1. `/accounts/cabinet/` — Личный кабинет

Композиция сверху вниз:

**5.1.1. Header страницы.** Заголовок «Личный кабинет», подзаголовок-описание, справа кнопки `Скачать счёт-акт` (заглушка, открывает toast «Раздел в разработке») и `Пополнить баланс` (`tms-btn-primary`, ведёт на `billing:deposit`).

**5.1.2. Алерт-бар про задолженность.** Показывать только если `billing_account.balance < billing_warn_threshold` (значение есть в контекст-процессоре `billing_account`). Текст: «Баланса хватит до &lt;дата&gt;. Пополните минимум на &lt;X&gt; ₽». Расчёт даты и суммы — в новом методе `BillingForecastService.estimate_runout(account)`. Если метода ещё нет — заглушка: «Баланс ниже порога, пополните» + кнопка. Реализацию калькулятора — отдельной задачей (см. §8).

**5.1.3. Hero-блок «Баланс + прогноз».** Тёмный градиентный блок, две колонки. Слева — крупно баланс из `billing_account.balance` + 3 кнопки (`Пополнить`, `Автоплатёж` (заглушка), `История операций` → `billing:transactions`). Справа — карточка прогноза с разбивкой: тариф, овердрафт по рейсам, модули, итог. Данные брать из `SubscriptionView.get_context_data` (`forecast_total`, `overage_fee`, `forecast_modules_fee`, `subscription.effective_monthly_price`). Чтобы переиспользовать на cabinet, вынести расчёт в `billing/services/forecast.py` и вызывать в обеих view.

**5.1.4. KPI-полоса.** 4 плитки в одну строку (на узких экранах — 2×2):
- **Рейсы за период** — `trip_usage["confirmed"]`. Дельта к прошлому периоду (если бэкенд не готов — спрятать дельту, показать только число). Спарклайн — заглушка SVG с захардкоженной полилинией ИЛИ пустой блок (выбор: показать пустую плитку без графика — это честнее).
- **Путевые листы** — `Waybill.objects.for_account(account).filter(...).count()` за период. Если расчёт тяжёлый — кэш на 5 мин через `cache.get_or_set`.
- **ТС активные / лимит** — `org_usage` стиль: `Vehicle.objects.for_account(account).filter(is_active=True).count()` + лимит из `subscription.plan` (но в текущей модели лимита на ТС нет — показать без лимита, просто число; добавление лимита ТС — отдельная задача).
- **Средний чек рейса** — заглушка («—») с подписью «скоро». Бэкенд не готов.

**Принцип:** где данных нет — честная заглушка, никакого моков с цифрами. Заглушку оформлять стилистически (приглушённый цвет, шрифт «Скоро», без графика).

**5.1.5. Профиль и команда.** Сохраняем существующий блок текущего `cabinet.html` про данные пользователя (ФИО, e-mail, роль) и список пользователей аккаунта (`users_in_account` в контексте). Стилизуем под новый язык — карточка с заголовком «Профиль и команда», таблица пользователей с аватаркой-инициалами, чипом роли, кнопкой `+ Пригласить` (если `can_manage_users`). Действия (создать/удалить/сбросить пароль) — оставить существующие view, поменять только обёртку.

**5.1.6. Свои организации.** Блок-карточка со списком `own_companies` из контекста. Кнопка `+ Добавить компанию` ведёт на `organizations:create?own=1` (как сейчас в org-dropdown).

**5.1.7. Безопасность аккаунта.** Карточка с тремя строками-настройками:
- 2FA — тоггл, **заглушка** (`disabled`, подпись «Скоро»). Бэкенда нет.
- Уведомления о входе на почту — тоггл, **заглушка**.
- IP-ограничения — кнопка «Настроить» → toast «Скоро».

Бэкенда для этих настроек нет — оставляем UI-каркас, чтобы потом подключить.

**5.1.8. Активные сессии.** Блок-карточка. Список из `UserSession.objects.filter(user=request.user, is_active=True)` (если модель есть в `accounts/models.py`; если ещё нет полей user_agent/ip — выводить только дату/время и метку «эта сессия» по `request.session.session_key`). Кнопка «Завершить» рядом с каждой — POST на новый view `accounts:terminate_session` (если такого нет — это ОТДЕЛЬНАЯ ЗАДАЧА, в этом ТЗ только UI и заглушка `disabled`).

### 5.2. `/billing/subscription/` — Мой тариф

Композиция сверху вниз:

**5.2.1. Header.** Заголовок «Мой тариф», подзаголовок «Управляйте подпиской, лимитами и модулями».

**5.2.2. Текущий тариф.** Карточка по макету: бейдж-плашка с буквой плана, название (`subscription.plan.name`), цена (`subscription.effective_monthly_price` ₽/мес), дата следующего списания (`subscription.current_period_end`), статус-чип «активен / триал / просрочен» исходя из `subscription.status` и `is_trial`. Справа кнопка `Сменить тариф` — открывает существующую модалку `#plan-change-modal` (логика уже есть в `subscription.js`, просто перевешиваем стили).

Для `corporate` — кнопка `Связаться с менеджером` (mailto, как сейчас).

Если есть `subscription.scheduled_plan` — показывать инфо-плашку «После &lt;дата&gt; перейдёте на &lt;план&gt;» с кнопкой «Отменить переход».

**5.2.3. Лимиты текущего периода.** Список из 5 строк (по макету):
- Рейсы — `trip_usage` (есть)
- Свои организации — `org_usage` (есть)
- Транспортные средства — **нет в контексте**, добавить в `SubscriptionView.get_context_data` через `get_vehicle_usage(account)` (написать сервис аналогично `get_organization_usage`); если у плана нет лимита ТС — показать без прогресс-бара, формат «18 ТС».
- Пользователи — `user_usage` (есть)
- Хранилище — **заглушка** (без бэкенда). Показать строку «Хранилище документов» с прочерком и подписью «расчёт скоро».

Каждая строка: иконка-плашка слева, название + чип статуса (`овердрафт` / `близко к лимиту` / нет), описание, текущее/лимит справа, прогресс-бар. Цвета бара: норма (синий-зелёный), `is-warn` ≥ 80%, `is-danger` > 100%. Если `limit is None` — показать «N / ∞» без бара.

**5.2.4. Подключённые модули.** Список из `Module.objects.filter(is_active=True)` + статус подключения для текущего аккаунта (`AccountModule`). Каждый модуль — строка с иконкой, названием, ценой, тогглом.
- `contracts` — реальный, бесплатный (0 ₽/мес из БД), включается через AJAX (вьюхи подключения/отключения модулей: если их нет — вернуть в этом ТЗ только UI, тоггл `disabled` с подписью «Подключение скоро»; если есть — использовать).
- Любые другие модули, появившиеся в БД, отрисовать автоматически.
- **НЕ выдумывать модули, которых нет в `Module`** (никаких «SMS-уведомлений» и «ЭДО» из макета). Если модулей всего один (`contracts`) — показать его + блок-плашку «Скоро появятся новые модули» и список планируемых (Petrolplus, ЭДО) как анонс без тогглов.

**5.2.5. Сравнение тарифов.** Сетка из 4 карточек ровно по тарифной сетке БД (см. §5.3). Каждая карточка — `name`, `monthly_price`, список лимитов (рейсы/организации/пользователи), CTA. Логика CTA:
- Текущий план → кнопка `disabled` «Текущий тариф», карточка с ободком `lk-plan--current`.
- Дороже текущего → «Перейти (апгрейд)», action — `subscription_upgrade` (есть).
- Дешевле текущего → «Перейти с &lt;дата конца периода&gt;», action — `subscription_schedule_downgrade` (есть).
- `corporate` (is_custom=True) → «Связаться с менеджером» (mailto).

Источник данных — `available_plans` уже в контексте + `subscription.plan` для подсветки текущего. Для подсветки текущего его нужно ВКЛЮЧИТЬ обратно в список — поменять `available_plans` или добавить `all_plans = Plan.objects.filter(is_active=True).order_by("display_order")` и в шаблоне рисовать его, а сравнивать с `subscription.plan_id`.

Цены — из `Plan.monthly_price` (НЕ хардкодить в шаблоне).

**5.2.6. Способ оплаты + реквизиты.** Две колонки:
- Левая — карта (CloudPayments). Если у аккаунта есть сохранённый токен → показать last4 + дату; если нет → кнопка «Привязать карту для автоплатежа» (заглушка, бэкенда сохранения токена пока нет — пометить «Скоро»).
- Правая — реквизиты для безналичной оплаты: «Плательщик», «ИНН/КПП», «Назначение: Пополнение баланса аккаунта #&lt;account.pk&gt;», НДС. Берём из `current_org` или `account` (если у `Account` есть собственные реквизиты — иначе подставлять текст «—» и кнопку «Указать реквизиты»). Кнопки: `Скачать счёт` (заглушка, форма генерации счетов на пополнение бюджета — отдельная задача), `Копировать реквизиты` (JS, copy-to-clipboard).

**5.2.7. История операций.** Последние 5 транзакций из `BillingTransaction.objects.filter(account=account).order_by("-created_at")[:5]`. Для каждой — иконка приход/расход, название (на основе `kind` / `type`), описание из `metadata` (если есть human-readable breakdown), сумма, баланс после (`balance_after`). Ссылка «Все операции →» на `billing:transactions`.

## 6. Требования к UI/реализации

### 6.1. Стилевые правила (соблюдать `docs/ui-guide.md`)

- Кнопки — `.tms-btn` + модификатор. **Одна** `.tms-btn-primary` на видимом экране (наверху страницы — «Пополнить баланс»). Остальные — `secondary`/`light`/`ghost`.
- Никаких инлайн `<style>` и `<script>` в шаблонах.
- Django-данные в JS — только через `data-*`-атрибуты на корневом элементе (паттерн уже использован в `subscription.html`, продолжаем).
- Flash-сообщения — только `messages.*`, дублировать `{% if messages %}` в шаблоне нельзя.
- `disabled` на кнопках — нативный атрибут, не CSS-класс.

### 6.2. CSS-токены (`static/css/lk-tokens.css`)

Завести `--lk-*` переменные для радиусов, теней, фоновых оттенков и компонентов карточек/чипов/прогресс-баров. Цветовые переменные — НЕ дублировать с `globals.css`, а ссылаться на них там, где совпадают (брендовый синий уже есть). Тёмный hero — только локально на странице ЛК, не глобально.

### 6.3. Адаптивность

Контентная сетка: 2 колонки на десктопе (≥1100px), 1 колонка на планшете/мобиле. KPI-полоса — 4 → 2 → 1. Сетка тарифов — 4 → 2 → 1. Никаких горизонтальных скроллов.

### 6.4. Доступность

- Все интерактивные элементы — `<button>`/`<a>` с `aria-label` где нет текста.
- Тоггл-свитчи — `<input type="checkbox">` внутри `<label>` (нативная семантика).
- Прогресс-бары — добавить `role="progressbar"`, `aria-valuenow`, `aria-valuemax`.
- Контраст текст/фон ≥ 4.5:1.

### 6.5. JS

Логика смены тарифа (модалка `#plan-change-modal`, AJAX к `subscription_upgrade` / `_schedule_downgrade` / `_cancel_downgrade`, обработка 402 «недостаточно средств») — **сохранить как есть**, только переписать селекторы под новые классы. Файл `static/billing/js/subscription.js` рефакторим, не переписываем с нуля.

Новые JS-функции:
- Copy-to-clipboard для реквизитов (одна функция, ~20 строк, native `navigator.clipboard`).
- Открытие/закрытие выпадашки уведомлений (заглушка-меню).
- Toast-уведомления для заглушек («Раздел в разработке») — если в проекте уже есть утилита для toasts через `messages` или `flash-wrap`, использовать её, иначе — короткий хелпер в `base.js`.

## 7. Tenant-изоляция и безопасность

- Все запросы в новых view-методах — через `Model.objects.for_account(account)`.
- `LoginRequiredMixin` — обязателен на cabinet и subscription view.
- В `BillingTransaction.objects.filter(account=account)` фильтр по аккаунту обязателен.
- При добавлении `terminate_session` (если будем добавлять в этом ТЗ — нет, в следующем) — проверка `session.user == request.user`.
- Никаких новых `@csrf_exempt`. AJAX-запросы — с CSRF-токеном из cookie (паттерн уже используется в `subscription.js`).

## 8. Что вынести в отдельные задачи (НЕ в этом ТЗ)

1. **Бэкенд расчёта runout-даты** (когда кончится баланс) — `BillingForecastService.estimate_runout()`. Сейчас в алерте используем заглушку.
2. **Лимит на количество ТС в плане** — нужно добавить поле `vehicle_limit` в `Plan`, миграция, обновить planned data.
3. **Управление сессиями** — модель `UserSession` с user_agent/ip, view `terminate_session`, `terminate_all_others`.
4. **2FA, IP-ограничения, уведомления о входе** — отдельная фича.
5. **Сохранение токена карты CloudPayments** для автоплатежа — отдельная фича, требует доработки CloudPayments-интеграции.
6. **Generation счетов на пополнение баланса** (PDF-счёт юрлицу) — отдельная задача.
7. **Уведомления (notification feed)** — модели и сервис.
8. **Спарклайны и дельты в KPI** — после того как накопим данные за >2 периода.
