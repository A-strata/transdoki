## 1. Цель и контекст

Текущая реализация блока «Профиль и команда» в `accounts/templates/accounts/cabinet.html` использует табличную вёрстку с инлайн-редактированием ФИО прямо в ячейках. UX перегружен: на каждую строку — четыре колонки с переключателями режима, строка «Редактировать → Сохранить» конкурирует за внимание со сбросом пароля, селект роли и управление ФИО находятся в одной группе кнопок.

Предыдущий редизайн ЛК (`docs/cabinet-redesign-spec.md`) эту карточку намеренно оставил в стороне — теперь пришло время.

Задача — переверстать блок под модель «профиль владельца отдельно + команда списком с kebab-меню» и перевести критичные действия (сброс пароля, смена ФИО, приглашение) в модалки с честным одноразовым показом временных паролей.

**Что уже есть на бэкенде** (не трогаем):
- `AccountUserCreateView` (`accounts:user_create`) — POST возвращает JSON `{ok, username, temp_password, full_name, role}` при `X-Requested-With: XMLHttpRequest`.
- `AccountUserPasswordResetView` (`accounts:user_reset_password`) — POST, JSON `{ok, username, temp_password, full_name}`. Запрещает сброс владельцу и самому себе.
- `AccountUserUpdateView` (`accounts:user_update`) — AJAX-only POST, обновляет `first_name`, `last_name` и `role` (роль — только чужому профилю и только `{admin, dispatcher, logist}`). Возвращает `{ok, first_name, last_name, role_display}`.
- Контекст `AccountCabinetView`: `account`, `profile`, `users_in_account`, `can_manage_users`.

Всё это переиспользуем — **новых endpoint'ов не заводим**, нового Python-кода в `accounts/` не пишем.

## 2. Скоуп

**Что в скоупе:**
- Переверстать блок «Профиль и команда» в `accounts/templates/accounts/cabinet.html`.
- Единый список команды (`lk-team-list`) — свой профиль первой строкой с классом `is-self`, остальные сотрудники — следом.
- Роль в строке — read-only inline-чип (`lk-role-chip`) рядом с ФИО. **Без** инлайн-селекта.
- Реализовать 5 модалок (invite, edit-self, **edit-user**, reset-confirm, reset-success) как описано в §5.
- Смена роли — внутри `#edit-user-modal` вместе с ФИО, одной транзакцией.
- Реализовать универсальный компонент credential card с кнопками копирования.
- Компонент тостов (`#toast-wrap` + JS `showToast()`), переиспользуемый.
- Обновить CSS (`static/accounts/css/cabinet.css`) и JS (`static/accounts/js/cabinet.js`) — см. §7.

**Что вне скоупа:**
- Отключение пользователей, 2FA, управление сессиями, уведомления о входе — показываются как заглушки с бейджем `lk-soon` и `disabled`-атрибутом.
- Смена собственного пароля владельца — кнопка «Сменить пароль» в self-card ведёт на существующий `PasswordChangeView` (Django built-in), отдельной модалки **не** делаем.
- Смена email владельца — поле `disabled` в модалке редактирования своего профиля, подсказка «через поддержку».
- Изменения в `accounts/views.py`, `accounts/services.py`, `accounts/urls.py` — **запрещены**. Если по ходу реализации выяснится, что чего-то не хватает на бэкенде, остановиться и доложить.

## 3. Структура нового блока

Единая карточка `lk-card` с заголовком «Профиль и команда». Справа в шапке — primary-action «+ Пригласить» (только если `can_invite=True`; при исчерпанном лимите — ссылка «Увеличить лимит»). Подзаголовка «Сотрудники» нет — список начинается сразу под шапкой.

### 3.1. Единый список (`lk-team-list`)

Одна лента строк. Первая строка — текущий пользователь (с классом `is-self`), далее — остальные сотрудники в порядке создания (или сортировки, выбранной на бэкенде; не принципиально).

Каждая строка (`lk-team-row`) — CSS-grid из **трёх** колонок:
`[avatar 32] [name + мета 1fr] [actions auto]`

Никаких отдельных колонок под статус и роль. И статус, и роль — словами внутри `.lk-team-info`.

#### 3.1.1. Обычная строка сотрудника

1. **Аватар** — инициалы 32×32.
2. **`lk-team-name`** — только ФИО, без чипов.
3. **`lk-team-info`** одной строкой:
   `{email или логин} · {роль словом, локализованная} · {статус словом}`
   - Статус `активен` пишется обычным цветом, как часть мета-строки. Не цепляет внимание.
   - Статус `ожидает входа` оборачивается в `<span class="lk-meta-warn">…</span>` и подсвечивается `--warning-text`.
   - Статус `отключён` оборачивается в `<span class="lk-meta-off">…</span>` (приглушённо-серый, без рамки).
   - Роль — обычным цветом мета-строки (`administrator` → `администратор`, и т.д.). Локализованная форма с маленькой буквы.
4. **`lk-actions-cell`** — kebab (`button.lk-kebab`) с меню:
   - «Редактировать…» → открывает `#edit-user-modal`, передаёт через `data-*` атрибуты: `profile-id`, `first-name`, `last-name`, `full-name`, `login`, `role`.
   - «Сбросить пароль» (`.is-warn`) → открывает `#reset-confirm-modal`, передаёт `profile-id`, `full-name`, `login`, `role-display`.
   - divider.
   - «Отключить пользователя» — `disabled` + бейдж `lk-soon`.

#### 3.1.2. Self-row (первая строка, `.is-self`)

Тот же каркас grid'а, но с поправками:

Эмфаз — структурный, без подложки:
- Аватар 40×40.
- `.lk-team-name { font-weight:700; font-size:var(--text-base) }`.
- Чёткое визуальное отделение от команды — `border-top:2px solid var(--border)` у первой строки команды (`.lk-team-row.is-self + .lk-team-row`).
- **Никакого чипа «это вы», никакого подстрочника, никакой подложки.**

Состав self-row:
1. Аватар 40×40.
2. `lk-team-name` — ФИО + `lk-role-chip` с текущей ролью (для владельца — `lk-role-chip--owner` брендовым цветом; для остальных ролей — нейтральный `lk-role-chip`). Это **исключение** для self-row: чип роли остаётся, потому что у владельца это смысловой акцент «это твой уровень доступа».
3. `lk-team-info` — `{email} · последний вход {request.user.last_login|date}`. Без статуса — очевидно, что вы в системе.
4. `lk-actions-cell` — kebab-меню (как у team-row, для единообразия). Пункты:
   - «Редактировать профиль…» (`<button type="button" class="lk-kebab-item" data-modal-open="edit-self-modal">`) → открывает `#edit-self-modal`.
   - «Сменить пароль» (`<a class="lk-kebab-item" href="{% url 'password_change' %}">`) → ссылка на встроенный Django-view.

   Важно: в self-kebab только 2 пункта, без divider'а и «Отключить пользователя (скоро)». Чип `lk-soon` в self-row не появляется.

### 3.2. Футер карточки

Под списком, мелким приглушённым текстом, **одно предложение**. Модель Billing v2 не начисляет ₽/сутки за overage по пользователям — при достижении лимита блокируется создание (см. `BillingProtectedMixin`). Текст футера должен это отражать честно:

> «{user_count_current} из {user_limit} пользователей по тарифу «{subscription.plan.name}». При превышении лимита создание новых сотрудников блокируется — увеличьте лимит в тарифе.»

Если лимит не задан (`user_limit is None`): «{user_count_current} пользователей в команде. На вашем тарифе ограничения нет.»

Если тарифа нет (бесплатный/дефолтный без подписки): просто «{user_count_current} пользователей в команде.»

Подстрочник про «сброс пароля показывает временный пароль один раз» — **убираем** (это инструкция к flow, она появится в самой модалке сброса).

**Важно:** `overage_per_user` как контекст-переменная **не нужен** и в `AccountCabinetView.get_context_data` не добавляется. Предыдущие версии брифа просили взять его из `billing/constants.py` — это был артефакт legacy-посуточного биллинга (`DAILY_RATE_USER`), который удаляется в итерации 6 и к Billing v2 отношения не имеет.

## 4. Права и видимость элементов (матрица)

| Действие | Владелец над собой | Владелец над другими | Админ над собой | Админ над другими (не-владелец) | Админ над владельцем | Диспетчер/Логист |
|---|---|---|---|---|---|---|
| Пригласить сотрудника | ✓ | ✓ | — (кнопка скрыта) | — | — | — |
| Изменить своё ФИО | ✓ (edit-self) | — | ✓ (edit-self) | — | — | ✓ (edit-self) |
| Редактировать другого (ФИО+роль в одном модале) | — | ✓ | — | ✓ | ✗ (пункта меню нет) | — (kebab не показываем) |
| Сбросить пароль другому | — | ✓ (кроме owner) | — | ✓ | ✗ | — |
| Отключить пользователя | — | (скоро) | — | (скоро) | (скоро) | — |

**Правило в шаблоне:** kebab-меню у строки команды рендерится **только если** `can_manage_users` **и** `p.user_id != request.user.id` **и** `p.role != "owner"`.

В `#edit-user-modal` поле «Роль» рендерится:
- для обычных сотрудников — `<select>` с опциями `{admin, dispatcher, logist}`;
- если каким-то образом модалка открыта для владельца (защита от ошибки вёрстки) — селект заменяется на read-only чип и кнопка «Сохранить» блокирует изменение роли.

У диспетчера/логиста `can_manage_users=False`, значит self-row отрисовывается как обычно (они меняют своё ФИО через `edit-self`), а у чужих строк не показывается ни kebab, ни какие-либо другие экшены.

## 5. Модалки — детальная UX-спецификация

Все модалки строятся по схеме из мокапа: overlay (`lk-modal-overlay`) + внутренний блок (`lk-modal`) с header (заголовок + крестик), body, footer (`lk-modal-actions`). ESC и клик по overlay закрывают модалку. Focus trap — обязателен, первый input получает фокус при открытии.

### 5.1. `#invite-modal` — приглашение сотрудника

Двухшаговая (состояния переключаются через `[data-modal-step="form"]` / `[data-modal-step="success"]`).

**Шаг 1 — форма:**
- Поле «Имя» (`first_name`, required).
- Поле «Фамилия» (`last_name`, required).
- Select «Роль» с опциями admin/dispatcher/logist (владелец не назначается через UI).
- Info-баннер про лимит тарифа: «На тарифе *{plan.name}* ваш лимит — N пользователей. Сейчас активны M, после приглашения будет M+1 из N». Если `M+1 == N` — баннер оранжевый (предпоследнее место). Если `M == N` — модалку не показываем, кнопка «+ Пригласить» неактивна, а рядом ссылка «Увеличить лимит → тариф».
- Primary-кнопка «Создать сотрудника» → POST на `{% url 'accounts:user_create' %}` с `X-Requested-With: XMLHttpRequest`.

**Обработка ответа:**
- `ok: true` → переключаем на шаг 2.
- `ok: false` со статусом 400 → вывести поле с ошибкой под соответствующим input'ом (ключи из `errors`), первая ошибка — также в summary наверху формы (см. `docs/ui-guide.md` §12).
- Статус 402 (лимит) → показать ошибку общим баннером.
- Статус 403 (нет прав) → закрыть модалку + flash.

**Шаг 2 — credential reveal:**
- Иконка-галочка в круге (`lk-success-ico`).
- Заголовок «Сотрудник создан», подзаголовок с ФИО и ролью.
- Credential card (`lk-cred`): две строки — «Логин» и «Пароль», у каждой кнопка «Копировать».
- Warning-баннер: «Пароль показан только один раз…»
- Единственная primary-кнопка «Готово» → закрывает модалку + `location.reload()` (чтобы новый сотрудник появился в списке) ИЛИ (лучше) клиентское добавление строки с данными из JSON-ответа.

### 5.2. `#edit-self-modal` — редактирование собственного профиля

Одношаговая.
- Поля: `first_name`, `last_name` (редактируемые).
- `email` — disabled input с хинтом «Email — логин для входа. Для смены обратитесь в поддержку».
- Блок «Роль» — read-only `lk-role-chip` с текущей ролью пользователя (для владельца — `lk-role-chip--owner`) + подсказка «Меняется только через поддержку».
- Primary-кнопка «Сохранить» → POST на `{% url 'accounts:user_update' request.user.profile.id %}` (actor сам себе) с `first_name`, `last_name`. Роль **не** отправляем (бэкенд всё равно игнорирует для `is_own_profile`).
- После `ok: true` — закрыть модалку, обновить ФИО в self-row, показать тост «Профиль обновлён».

### 5.3. `#edit-user-modal` — редактирование сотрудника (ФИО + роль)

Одношаговая. Заменяет бывшую `#edit-name-modal` из версии 1.0.

- Открывается из kebab → «Редактировать…». Перед открытием JS заполняет поля из `data-*` атрибутов kebab-пункта: `data-profile-id`, `data-first-name`, `data-last-name`, `data-login`, `data-role`.
- Заголовок: «Редактировать пользователя».
- Подсказка вверху: «Логин изменить нельзя. Роль сохраняется вместе с ФИО — одним подтверждением.»
- Поля:
  - `first_name` (required).
  - `last_name` (required).
  - `role` — `<select>` с `{admin, dispatcher, logist}`, выбор из `data-role`. Под селектом — `lk-field-hint`: «Владелец меняется только через поддержку».
  - `login` — disabled input (чтобы было видно, чей профиль редактируем).
- Primary-кнопка «Сохранить изменения» → один POST на `{% url 'accounts:user_update' profile_id %}` с `first_name`, `last_name`, **`role`**.
- Ответ:
  - `ok: true` → закрыть модалку, обновить в строке команды ФИО, инициалы аватара и `lk-role-chip`, тост: «Пользователь обновлён». Если роль поменялась — тост: «Роль обновлена: {role_display}».
  - `ok: false` с полевыми ошибками → показать под соответствующими input'ами и в summary наверху формы.
  - 403 → закрыть модалку + flash «Недостаточно прав».
- **Откат / undo** не предусмотрен: действие явное, подтверждено в модалке. Если пользователь передумал — просто открывает модалку ещё раз и меняет обратно.

### 5.4. `#reset-confirm-modal` — подтверждение сброса пароля

Одношаговая. Открывается из kebab «Сбросить пароль».
- Заполняется из `data-profile-id`, `data-full-name`, `data-login`.
- Текст: «Будет сгенерирован новый временный пароль для **{full_name}**. Текущий пароль сразу перестанет работать — все открытые сессии пользователя завершатся.»
- Warning-баннер: «Новый пароль будет показан один раз — после закрытия окна восстановить его нельзя, только сбросить снова.»
- Primary-кнопка «Сбросить пароль» (красноватая — `tms-btn-primary` достаточно, но заголовок и warning-баннер дают визуальный контекст) → POST на `{% url 'accounts:user_reset_password' profile_id %}`.
- При `ok: true` — закрыть эту модалку, открыть `#reset-success-modal` с данными из ответа.
- При `ok: false` — показать сообщение в body модалки, не закрывать.

### 5.5. `#reset-success-modal` — credential reveal после сброса

Одношаговая.
- Иконка-галочка + заголовок «Пароль сброшен», подзаголовок с ФИО и ролью.
- Credential card: логин и новый временный пароль, кнопки «Копировать» на обеих строках.
- Warning-баннер про одноразовый показ.
- Primary-кнопка «Готово» → закрывает модалку.
- На странице обновлять ничего не нужно — пароль в списке команды мы не показываем.

## 6. Смена роли (внутри модалки)

Инлайн-селект, instant-save и toast с undo **удалены** в версии 1.1.

Логика теперь:
1. Пользователь жмёт kebab → «Редактировать…» → открывается `#edit-user-modal`.
2. JS заполняет форму из `data-*` атрибутов пункта меню (включая текущий `role`).
3. Пользователь меняет ФИО и/или роль и жмёт «Сохранить изменения».
4. Один AJAX-POST на `{% url 'accounts:user_update' profile_id %}` с полями `first_name`, `last_name`, `role`.
5. На `ok: true` — обновить в строке: ФИО, инициалы аватара, `lk-role-chip` (текст и, если роль = `owner`, модификатор `--owner` — впрочем, owner через UI не назначается, поэтому в реальной жизни модификатор для строки команды не пригодится).
6. На `ok: false` — показать ошибки в форме, **не** закрывать модалку.

**Бэкендных изменений не нужно:** `AccountUserUpdateView` уже принимает `first_name`, `last_name` и `role` в одном POST.

Тосты остаются как глобальный механизм для confirmations («Профиль обновлён», «Пароль сброшен», ошибки). Поведение — без undo.

## 8. Tenant-изоляция и безопасность

- В шаблоне не рендерить экшены для строк, где `p.user_id == request.user.id` или `p.role == "owner"` (дублирует бэкенд-проверки).
- В JS не доверять `profile-id` из `data-*` — передавать как есть, бэкенд провалидирует через `UserProfile.objects.filter(account=account)`.
- Временные пароли показывать **только** в ответе текущего запроса — не сохранять в `data-*`, не логировать в консоль, не отправлять в аналитику. После закрытия модалки переменная должна быть сборщиком мусора подметена.
- Все AJAX-POST — с CSRF-токеном. Брать из `{% csrf_token %}` в скрытой форме в теле страницы (одной на все модалки) или из `document.cookie['csrftoken']` + заголовок `X-CSRFToken`. Второй способ — предпочтительнее.
- `navigator.clipboard.writeText` — обязательно fallback на textarea + execCommand для старых браузеров (Safari <13.1).

## 9. Контекст, который нужно подкинуть в шаблон

`AccountCabinetView.get_context_data` уже даёт `account`, `profile`, `users_in_account`, `can_manage_users`. Добавить:

- `user_limit` (int | None) — из `subscription.effective_user_limit` (если подписка есть и not None); иначе `None` → рендерим «без ограничения». **Уже в контексте.**
- `user_count_current` — `users_in_account.count() + 1` (сам пользователь, т.к. `users_in_account` исключает `request.user` через `.exclude(user=request.user)`). **Уже в контексте.**
- `can_invite` — bool: `can_manage_users and (user_limit is None or user_count_current < user_limit)`. **Уже в контексте.**
- `last_activity` — аннотация на `users_in_account` через `annotate(last_activity=Max("user__sessions__last_activity"))`. **Уже сделано** в `get_queryset`.

`overage_per_user` в контекст **не добавляется** (см. §3.2 — Billing v2 не использует посуточный overage по пользователям).

`users_in_account` оставляем с `.exclude(user=request.user)` — self-row рендерится отдельным блоком в шаблоне **перед** `{% for p in users_in_account %}`, в том же `.lk-team-list`. Это сохраняет тестируемость queryset'а и не требует переупорядочивания.

## 11. Acceptance criteria

1. На странице `/accounts/cabinet/` блок «Профиль и команда» визуально соответствует референсу из `cabinet-mockup.html` v1.3.
2. Self-row — первая строка `.lk-team-list` с классом `is-self`, без чипа «это вы», без подстрочника, без подложки.
3. Self-row отделён от остальных строк линией `border-top:2px solid var(--border)` у первой строки команды; ФИО `700/text-base`, аватар 40×40.
4. Self-row рендерится для любой роли (владелец/админ/диспетчер/логист). Actions — **kebab** с двумя пунктами: «Редактировать профиль…» (→ `#edit-self-modal`) и «Сменить пароль» (→ `{% url 'password_change' %}`). Inline-кнопок на self-row нет.
5. У сотрудников команды роль и статус — словами в мета-строке `.lk-team-info`. Чипа роли в строке команды нет. Чип `lk-role-chip--owner` есть только в self-row у владельца.
6. Слово «ожидает входа» в мета-строке подсвечено `lk-meta-warn`. Слово «отключён» — `lk-meta-off`. Слово «активен» — обычным цветом мета.
7. «+ Пригласить» виден только при `can_invite=True`. При достижении лимита — кнопка заменена на ссылку «Увеличить лимит».
8. Kebab у team-row содержит ровно три пункта: «Редактировать…», «Сбросить пароль», «Отключить пользователя (скоро)». Kebab у self-row содержит ровно два пункта: «Редактировать профиль…» и «Сменить пароль» (без divider'а и «Отключить»).
9. Kebab → «Редактировать…» открывает `#edit-user-modal` с ФИО + ролью; «Сохранить изменения» → один AJAX-POST на `accounts:user_update`, без перезагрузки.
10. Kebab → «Сбросить пароль»: confirm → success показывает новый пароль один раз, он копируется в буфер по кнопке.
11. «+ Пригласить» создаёт пользователя и показывает credential card во втором шаге той же модалки.
12. Пункт self-kebab «Сменить пароль» — это `<a class="lk-kebab-item" href="{% url 'password_change' %}">`, не `<button>`. Визуально неотличим от остальных пунктов (`.lk-kebab-item { text-decoration:none; color: inherit }`).
13. Для диспетчера/логиста kebab не отображается у чужих строк; self-row для них работает как обычно (можно править своё ФИО, свой kebab доступен всем ролям).
14. У строки владельца kebab отрисовывается только для не-владельцев — у строки самого владельца kebab-меню с «Редактировать…» / «Сбросить пароль» не отрисовывается. Исключение — self-row владельца: у неё свой self-kebab на 2 пункта (см. п. 4).
15. ESC и клик по overlay закрывают модалку; focus trap работает; при открытии фокус на первом input.
16. При открытии любой модалки все открытые kebab-меню закрываются; submit-кнопка сбрасывается из disabled-состояния при `closeModal` (регрессия «сброс работает только один раз» не воспроизводится).
17. Тосты не наслаиваются бесконечно — старый скрывается через 4.5 сек. Опция `undo` в `showToast` отсутствует.
18. Футер карточки — единое предложение про квоту/тариф/overage. Старого подстрочника про инструкцию по сбросу пароля **нет**.
19. CSP-совместимость: никаких inline-скриптов и стилей.
20. Никаких console.log / alert() в продакшн-коде.
21. Проходит `ruff check .`, `python manage.py test accounts`, `python manage.py test tests.test_tenant_isolation`.
22. Новых миграций нет. Новых зависимостей нет.
23. Playwright smoke-скрипт `scripts/cabinet_smoke.mjs` проходит: (а) открывает kebab → «Редактировать…» → меняет роль → сохраняет → в мета-строке появляется новая роль; (б) второй раз открывает kebab у той же строки → «Сбросить пароль» → confirm → success → видит модалку с паролем.
