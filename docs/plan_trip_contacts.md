# План: множественные контакты на точках рейса (TripPointContact)

> Статус: **запланировано**, не реализовано.
> Дата: 2026-03-30

## Контекст

Сейчас `TripPoint` хранит контакт в плоских полях `contact_name` + `contact_phone` (один контакт на точку). В реальности на точке погрузки/выгрузки может быть несколько контактов (логист, кладовщик, охрана).

## Решение

Новая модель `TripPointContact` в приложении `trips`:

```python
class TripPointContact(models.Model):
    point = models.ForeignKey(TripPoint, on_delete=CASCADE, related_name="contacts")
    name = models.CharField(max_length=150)
    phone = models.CharField(max_length=25, blank=True)
```

## Принципы

- **Снимок, не ссылка.** Данные копируются при создании рейса и не меняются при обновлении справочника `OrganizationContact`. Рейс — документ, фиксирует состояние на момент создания.
- **Без FK на OrganizationContact.** Источник контакта непредсказуем (заказчик, отправитель, получатель, ручной ввод). Мягкая ссылка не даёт ценности после создания рейса.

## Автоподстановка контактов в форме рейса

Диспетчер выбирает контрагентов рейса (заказчик, отправитель, получатель). JS берёт ID этих организаций из полей формы и запрашивает их контакты из справочника `OrganizationContact`:

```
GET /organizations/contacts/search/?org_ids=5,12,18
```

Эндпоинт возвращает контакты только этих трёх организаций с указанием, к какой организации относится каждый контакт. Диспетчер видит короткий релевантный список и выбирает нужного (данные копируются в снимок) или вводит вручную.

При смене организации в форме — список контактов автоматически обновляется.

## Сохранение ручного контакта в справочник

Если диспетчер ввёл контакт вручную (его нет в справочнике), под полями контакта появляется опция:

```
☑ Сохранить в контакты: [ООО «Альфа» (отправитель) ▾]
```

- По умолчанию предлагается очевидная организация: для точки погрузки — отправитель, для выгрузки — получатель.
- Через выпадающий список можно выбрать другого контрагента рейса (например, заказчика — когда заказчик даёт свои контакты, чтобы закрыть перевозчику прямой выход на отправителя/получателя).
- При сохранении рейса контакт создаётся и в `TripPointContact` (снимок), и в `OrganizationContact` выбранной организации (справочник).
- Если галочку не ставить — контакт остаётся только в рейсе. Справочник не засоряется разовыми контактами.

## Миграция данных

1. Создать модель `TripPointContact`
2. Миграция данных: для каждого `TripPoint` с непустым `contact_name` создать `TripPointContact`
3. Удалить поля `contact_name`, `contact_phone` из `TripPoint`
4. Обновить формы рейса (заменить два плоских поля на динамический список контактов)
5. Обновить `trip_detail.html` — отображение списка контактов вместо одного

## UI в форме рейса

- Под каждой точкой (погрузка/выгрузка) — список контактов с кнопкой «+ Добавить»
- При клике — autocomplete из контактов организаций-участников рейса
- Возможность ввести вручную (без привязки к справочнику)
- Каждый контакт — строка с name, phone и кнопкой удаления
- Под ручным контактом — галочка «Сохранить в контакты» с выбором организации
